# F2FS 闪存友好文件系统概述

## 〇、为什么需要 F2FS？

### 传统文件系统的问题

在 F2FS 出现之前，Linux 主流文件系统（ext4、XFS、Btrfs）都是为 HDD（机械硬盘）设计的：

**HDD 特性：**
- 随机写入需要寻道，代价昂贵（~10ms）
- 顺序写入性能远高于随机写入（100x 差距）
- 覆盖写入无额外代价
- 无磨损均衡问题

**传统文件系统优化策略：**
- 就地更新（in-place update）减少空间碎片
- 延迟分配（delayed allocation）聚合小写入
- 日志只记录元数据，数据直接覆盖写

### 闪存（SSD/eMMC/UFS）的挑战

闪存具有完全不同的特性：

| 特性 | HDD | 闪存 |
|------|-----|------|
| 随机读写 | 慢（需要寻道） | 快（无机械部件） |
| 覆盖写入 | 直接覆盖 | 需要先擦除整个块 |
| 写入粒度 | 512B/4KB 扇区 | Page（4KB-16KB） |
| 擦除粒度 | 无 | Block（128KB-4MB） |
| 寿命 | 机械磨损 | 擦写次数限制（P/E cycles） |
| 写入放大 | 无 | 有（1次逻辑写→多次物理写） |

**闪存的写入放大问题：**
```
传统文件系统更新 4KB 数据：
1. 读取整个擦除块（比如 512KB）到缓存
2. 修改缓存中的 4KB 数据
3. 擦除整个块
4. 写回整个 512KB

结果：4KB 逻辑写入 → 512KB 物理写入（128倍放大！）
```

## 1. F2FS 的解决方案

F2FS（Flash-Friendly File System）由三星在 2012 年开发，专门针对闪存特性优化：

### 核心设计：Log-structured File System (LFS)

F2FS 采用日志结构设计，**将所有写入转换为顺序追加**：

```
传统文件系统（就地更新）：
  文件A: [Block 100] → 更新 → [Block 100']（覆盖）
  
F2FS（日志结构）：
  文件A: [Block 100] → 更新 → [Block 500]（追加到日志末尾）
         ↓
      [Block 100] 变为无效，等待垃圾回收
```

### 解决的核心问题

| 问题 | F2FS 解决方案 |
|------|--------------|
| 写入放大 | 日志结构，只做顺序写，避免读-改-写 |
| 随机写性能 | 转换为顺序追加，利用闪存并行性 |
| 磨损均衡 | 自然的磨损分布，配合 FTL 做二级均衡 |
| 垃圾回收开销 | 冷热数据分离，减少有效数据迁移 |
| 小文件性能 | Inline data/dentry/xattr |
| 大文件顺序读 | Extent tree 缓存连续块映射 |

## 3. 磁盘布局

F2FS 将整个分区划分为 6 大区域：

```
┌──────────────┬────────────┬──────────┬──────────┬──────────┬──────────────┐
│  Superblock  │ Checkpoint │   SIT    │   NAT    │   SSA    │ Main Area    │
│   (SB)       │   (CP)     │          │          │          │              │
├──────────────┼────────────┼──────────┼──────────┼──────────┼──────────────┤
│              │            │ Segment  │  Node    │ Summary  │              │
│   元数据      │  一致性状态  │ Info     │ Address  │  Area    │   数据区域    │
│   文件系统     │  位图/游标   │ Table    │  Table   │ 索引摘要   │ (LFS日志)    │
│              │            │ 段有效块   │ 节点映射   │          │              │
└──────────────┴────────────┴──────────┴──────────┴──────────┴──────────────┘
     1 seg          2 seg       ~          ~          ~         剩余空间

Main Area 细分：
┌─────────────────────────────────────────────────────────────────┐
│  HOT_NODE │ WARM_NODE │ COLD_NODE │ HOT_DATA │ WARM_DATA │ COLD │
│  (目录)    │ (direct)  │ (indirect)│ (目录数据) │ (用户数据) │ (多媒体)│
└─────────────────────────────────────────────────────────────────┘
                        6 个并发写入日志
```

## 4. 核心概念

### 4.1 Node 与 NAT

F2FS 使用间接索引，所有文件数据通过 Node 寻址：

```
inode → direct node → data block
      ↘ indirect node → direct node → data block
        ↘ double indirect node → ...

NAT (Node Address Table)：
  Node ID → Block Address
  实现 Node 的逻辑地址到物理地址映射
```

### 4.2 Segment 与 Section

```
Block (4KB) → Segment (2MB = 512 blocks) → Section (多个 segments) → Zone

Segment 是分配和回收的基本单位
Section 用于多段并发写入
Zone 对应闪存的擦除块
```

### 4.3 Multi-head Logging

F2FS 维护 6 个并发日志，根据数据冷热程度分离：

| 日志类型 | 数据特征 | 更新频率 | 示例 |
|---------|---------|---------|------|
| HOT_NODE | 目录 inode | 极高 | /tmp, /var/log |
| WARM_NODE | 直接节点 | 高 | 普通文件 inode |
| COLD_NODE | 间接节点 | 低 | 大文件索引 |
| HOT_DATA | 目录数据 | 高 | 目录项 |
| WARM_DATA | 用户数据 | 中 | 文档、代码 |
| COLD_DATA | 多媒体 | 极低 | 视频、图片 |

### 4.4 Checkpoint

Checkpoint 保证文件系统一致性：
- 包含所有元数据的一致性快照
- 支持 roll-forward recovery
- 双 checkpoint 交替更新

## 5. 数据流示例

### 写入流程

```
1. 用户写入数据
   ↓
2. F2FS 判断数据冷热类型
   ↓
3. 选择对应的日志头（6个之一）
   ↓
4. 追加到当前段的末尾
   ↓
5. 更新 NAT（如果是新 node）
   ↓
6. 更新 SIT（段的有效块计数）
   ↓
7. 定期触发 Checkpoint 持久化元数据
```

### 读取流程

```
1. 通过 inode 找到 node ID
   ↓
2. 查询 NAT 获得 node 的物理地址
   ↓
3. 读取 node，获得数据块地址
   ↓
4. 如果命中 extent cache，直接返回
   ↓
5. 读取数据块
```

## 6. F2FS 优势总结

1. **闪存友好**：减少写入放大，延长设备寿命
2. **高性能**：多流并发写入，优秀的随机写性能
3. **低延迟**：避免 GC 引起的长尾延迟
4. **省电**：减少不必要的数据迁移
5. **特性丰富**：原子写、压缩、加密、在线磁盘碎片整理

## 7. 适用场景

F2FS 特别适合：
- Android 设备（Google 官方推荐）
- 嵌入式系统
- SD 卡、eMMC、UFS 存储
- 需要高随机写性能的场景
- 对功耗敏感的移动设备

不适合：
- 企业级 SSD（它们有强大的 FTL）
- 以顺序读为主的场景
- 需要快照功能的场景
